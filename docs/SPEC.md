# Техническое задание: flux-diff — CLI для локальной валидации Flux GitOps (Go)

## 1. Общие сведения

### 1.1. Назначение
Разработать CLI-утилиту на языке Go под названием **flux-diff**, повторяющую функциональность референса [flux-local](https://github.com/allenporter/flux-local), для локальной сборки и сравнения ресурсов Flux GitOps-репозитория. Утилита работает только с локальным git-репозиторием (без подключения к живому кластеру) и помогает повысить качество коммитов, PR и общего локального тестирования.

### 1.2. Референс
- Репозиторий: `https://github.com/allenporter/flux-local`
- Оригинал написан на Python; целевая реализация — на Go.

### 1.3. Форм-фактор поставки
Поставка осуществляется в виде **CLI-утилиты** (один исполняемый бинарь `flux-diff`), по аналогии с самим flux-local. Отдельная библиотека/SDK для встраивания в другие сервисы в объём работ не входит. Внутренняя организация кода должна быть модульной и тестируемой, но публичный стабильный API библиотеки не является требованием.

### 1.4. Принципиальные ограничения
- Анализируется **только локальный git-репозиторий**; обращений к работающему кластеру нет.
- **Секреты автоматически обнаруживаются и маскируются** — содержимое `data`/`stringData` заменяется на `"*** (SECRET) ***"`, в stderr выводится предупреждение с количеством секретов. Поддержка SOPS/age не требуется.
- Поддерживаются не все возможности Flux, но достаточно для типовых сценариев валидации.
- Git используется через **Go SDK** (`github.com/go-git/go-git/v5`). Внешний бинарник не требуется.
- Kustomize используется через **Go SDK** (не внешний бинарник).
- Helm используется через **Go SDK** (`helm.sh/helm/v4`). Внешний бинарник не требуется.
- Инфляция HelmRelease **всегда включена** (отдельный флаг не требуется).
- Namespace ресурсов берётся **из манифестов** Flux (отдельный флаг не требуется).

## 3. Функциональные требования

Реализуются **только** команды `build` и `diff`. Команды `get` и `test` из референса в объём работ **не входят**.

### 3.1. Команда `build` — сборка объектов
Аналог `flux-local build`. Под капотом — kustomize Go SDK + helm template.

Требования:
- Сборка всех `Kustomization`-объектов внутри git-репозитория по заданному пути (`--path`).
- Подкоманда для `Kustomization`: `flux-diff build ks [--path <path>]`.
  - Запускает kustomize build через Go SDK для каждого ресурса.
  - Автоматически выполняет инфляцию всех найденных HelmRelease (helm всегда включён).
- Подкоманда для `HelmRelease`: `flux-diff build hr <name> [--path <path>]`.
  - Инфляция через `helm template` поверх собранных манифестов.
  - Namespace берётся из манифеста HelmRelease.
- Вывод итогового YAML-набора манифестов в stdout.
- Многодокументный YAML на выходе должен быть валиден и пригоден для последующих инструментов.

### 3.2. Команда `diff` — сравнение изменений
Аналог `flux-local diff`. Локальный аналог `flux diff`.

Требования:
- Сравнение `Kustomization` (`flux-diff diff ks <name>`): выполняется kustomize build через Go SDK против текущего состояния локального репо и против предыдущей git-ревизии/ветки, затем выводится дифф.
- Сравнение `HelmRelease` (`flux-diff diff hr <name>`): инфляция Helm локально и показ диффов в результирующих объектах. Namespace берётся из манифеста.
- Поддержка указания ветки/ревизии для сравнения через флаг `--branch` / `-b`. Если не указан — автоматически определяется default branch через `git symbolic-ref refs/remotes/origin/HEAD` (fallback: `main`/`master`).
- **Цветная подсветка вывода**: удалённые строки/значения — красным, добавленные — зелёным, заголовки ханков — cyan (unified-diff). Подсветка реализуется через ANSI-escape-последовательности.
- Управление подсветкой: режимы `auto` (включать только при выводе в TTY), `always`, `never`. По умолчанию — `auto`; при перенаправлении вывода в пайп/файл цвета не добавляются. Поддержка переменной окружения `NO_COLOR` для отключения.
- Логика вычисления диффа отделена от форматтера раскраски: внутренне дифф формируется как «сырой» текст без ANSI, затем при необходимости раскрашивается. Это обеспечивает корректную работу в пайпах, файлах и PR-комментариях.
- Diff реализован через LCS-алгоритм (Longest Common Subsequence) с группировкой в ханки (hunks) unified diff формата.
- Корректная работа с overlay и несколькими кластерами (напр. `dev` и `prod`) — итоговый полностью собранный вывод.
- Код возврата процесса: `0` — успех (нет различий), `1` — есть различия, `2` — ошибка.

## 4. Требования к CLI и внутренней организации

### 4.1. Интерфейс командной строки
- Имя бинаря: `flux-diff`.
- Структура команд: `flux-diff <command> <resource> [args] [flags]`, где `command ∈ {build, diff}`, `resource ∈ {ks, hr}`.
- Флаги:
  - `-p, --path` — путь к кластеру/каталогу в репозитории.
  - `-b, --branch` — ветка/ревизия для сравнения (только для `diff`; по умолчанию — автоопределение default branch).
  - `--color {auto,always,never}` — управление подсветкой diff (только для `diff`).
- Вывод в stdout пригоден для пайпинга.
- Диагностические сообщения и логи — в stderr, чтобы не загрязнять stdout с манифестами/диффом.
- Поддержка `--help` для корневой команды и подкоманд.

### 4.2. Внутренняя организация кода
Структура пакетов:
```
cmd/flux-diff/main.go       — точка входа
internal/
  cli/                       — разбор аргументов CLI, команды build/diff
  diff/                      — вычисление и форматирование diff (LCS, ANSI-подсветка)
  flux/                      — парсинг и модель объектов Flux (Kustomization, HelmRelease, HelmRepository)
  git/                       — операции с git через go-git SDK (worktree, branch, revision)
  helm/                      — инфляция HelmRelease через Helm Go SDK
  kustomize/                 — kustomize build через Go SDK
```

- Чёткое разделение слоёв: разбор аргументов CLI → загрузка/парсинг git-репо → построение модели объектов (Kustomization/HelmRelease/…) → оркестрация инструментов (kustomize SDK/helm) → форматирование вывода.
- Внутренние пакеты тестируемы в изоляции (бизнес-логика отделена от слоя CLI и от вызовов внешних бинарников).
- Все внешние инструменты (git, kustomize, helm) работают через Go SDK — внешние бинарники не требуются.

### 4.3. Обработка ошибок и логирование
- Идиоматичные Go-ошибки (`error`), оборачивание с контекстом.
- Диагностические сообщения в stderr.
- Понятные сообщения при отсутствии внешних бинарников или невалидных манифестах.
- Осмысленные коды возврата процесса: `0` — успех, `1` — есть различия, `2` — ошибка.

### 4.4. Контекст и отмена
- Длительные операции выполняются с `context.Context` (таймауты, отмена по сигналу прерывания).

## 5. Интеграция с CI (GitHub Action и аналоги)

Опциональный, но желательный модуль — повторение CI-возможностей референса:
- **diff-action**: вычисление финальных диффов `Kustomization`/`HelmRelease` с поддержкой overlay и нескольких кластеров; вывод диффа пригоден для публикации в PR-комментарии.
- Поддержка матрицы (несколько `cluster_path` × несколько `resource`).
- Kustomize и Helm встроены в бинарь через Go SDK — внешние бинарники не требуются.

## 6. Нефункциональные требования

### 6.1. Тестируемость
- Юнит-тесты на парсинг и построение модели объектов.
- Юнит-тесты на diff-вычисление и форматирование (подсветка, режимы цвета).
- Интеграционные тесты с тестовыми данными кластера (`tests/testdata/cluster/`).
- Воспроизводимость: фикстуры тестовых репозиториев в составе проекта.

### 6.2. Производительность
- Параллельная обработка независимых `Kustomization`/`HelmRelease`, где это безопасно.
- Кэширование результатов `kustomize build`/`helm template` в рамках одного запуска при повторных обращениях.

### 6.3. Совместимость
- Поведение должно быть «достаточно близким» к Flux для типовых (в т.ч. домашних/home-lab) сценариев.
- Корректная работа с многокластерными репозиториями.

### 6.4. Безопасность
- Секреты (Secret resources) автоматически обнаруживаются в выходных данных.
- Значения полей `data` и `stringData` заменяются на `"*** (SECRET) ***"` — визуальная маркировка.
- В stderr выводится предупреждение: `⚠ N secret(s) found and redacted`.
- Поддержка SOPS/age **не требуется** — секреты маскируются без расшифровки.
- Никаких обращений к реальному кластеру и сетевых вызовов к API кластера.

### 6.5. Документация
- README с quickstart и примерами команд.
- Описание флагов и кодов возврата.
- Примеры CI-конфигураций.

### 6.6. Лицензия
- Apache-2.0 (как у референса), если иное не указано заказчиком.

## 7. Примеры использования

```bash
# Сборка всех Kustomization в кластере (с автоматической инфляцией HelmRelease)
flux-diff build ks --path clusters/prod/

# Инфляция конкретного HelmRelease (namespace берётся из манифеста)
flux-diff build hr podinfo --path clusters/prod/

# Дифф Kustomization против default branch
flux-diff diff ks apps --path clusters/prod/

# Дифф Kustomization против указанной ветки
flux-diff diff ks apps --path clusters/prod/ --branch main

# Дифф HelmRelease (namespace берётся из манифеста)
flux-diff diff hr podinfo --path clusters/prod/

# Дифф с принудительной подсветкой
flux-diff diff ks apps --path clusters/prod/ --color always
```
